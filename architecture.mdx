This page explains how HCIMS AI is organized as a monorepo, how data moves through the system from a raw guest review to a closed and verified ticket, and why each major technology was chosen. Read this before diving into configuration or feature guides.

## Repository structure

The project root is a monorepo. The two primary application directories have separate dependency trees and can be deployed independently.

```text
HCIMS-AI/
├── hcims-ai-main/                      # React + Vite frontend
│   ├── src/                            # UI pages, components, contexts, hooks
│   ├── supabase/                       # Edge functions and DB migrations
│   ├── package.json
│   └── .env.example
├── HCIMS-AI-Complaint-System-main/     # FastAPI backend + ML pipeline
│   ├── api.py                          # Prediction, admin, translation, proof endpoints
│   ├── src/                            # Prediction and proof validation logic
│   ├── data/                           # Training and input data
│   ├── mlops/                          # MLflow and DVC support files
│   ├── artifacts/                      # Trained model artifacts (generated at runtime)
│   ├── requirements.txt
│   └── .env.example
├── nonescape-master/                   # Nonescape reference used for proof validation setup
├── start.ps1                           # Windows dev startup helper
└── README.md
```

The frontend (`hcims-ai-main/`) never imports from the backend directory. All communication between the two happens over HTTP at runtime — the frontend calls `VITE_API_URL` (default `http://localhost:8001`) and Supabase endpoints.

## End-to-end data flow

The diagram below shows how a guest review becomes a closed ticket.

```text
Guest review CSV
       │
       ▼
[Admin uploads via UI]
       │
       ▼
FastAPI  /predict  endpoint
  ├── sentence-transformers embeddings
  ├── KMeans complaint/appreciation classification
  ├── Department assignment (Housekeeping, F&B, Engineering, IT, Front Office)
  └── Priority scoring (High / Medium / Low)
       │
       ▼
Supabase Postgres (complaints table, tickets table)
       │
       ▼
React frontend reads tickets via Supabase JS client
  ├── Admin sees all tickets across all departments
  └── Department staff see only their assigned tickets (RLS enforced)
       │
       ▼
Staff resolves ticket → uploads proof image
       │
       ▼
FastAPI  /complaints/{id}/resolve_with_proof  endpoint
   ├── Nonescape model scores image for AI generation probability
   ├── Rejects if confidence >= AI_IMAGE_CONFIDENCE_THRESHOLD (default 0.45)
   └── Stores accepted image in Supabase Storage "proofs" bucket
       │
       ▼
Ticket status updated to resolved in Supabase Postgres
       │
       ▼
Admin analytics dashboard reflects updated resolution data
```

## Key technology choices

**sentence-transformers for ML embeddings.** Guest reviews are encoded into dense vector representations using a pre-trained sentence-transformer model. Cosine similarity between a review embedding and reference complaint/appreciation embeddings drives the classification decision. This approach handles paraphrasing and informal language far better than keyword matching.

**scikit-learn KMeans for department routing.** After a review is classified as a complaint, a KMeans clustering model trained on labeled hotel complaint data assigns it to one of the five operational departments. The trained model artifacts (`artifacts/df.pkl` and related files) are generated by `src/train_model.py` on first run and reused on subsequent starts.

**Nonescape for proof validation.** The `nonescape-mini-v0.safetensors` model analyzes each uploaded proof image and returns a confidence score for AI generation. Images where `synthetic_confidence` meets or exceeds the configured threshold (`AI_IMAGE_CONFIDENCE_THRESHOLD`, default `0.45`) are rejected before the ticket is marked resolved. This closes a common abuse vector where staff could fabricate evidence.

**Sarvam AI for multilingual support.** Hotel operations staff in India often work in regional languages. Sarvam AI's translation and speech-to-text APIs, exposed through the backend, allow the UI to render in 22+ Indian languages and accept voice input. The `SARVAM_API_KEY` in the backend `.env` connects to this service.

**Supabase as the unified data platform.** Supabase provides four services under a single project: PostgreSQL for relational data, Auth for session management and JWT tokens, Storage for the `proofs` bucket, and Row-Level Security policies for data isolation. Using one platform for all four eliminates the operational overhead of maintaining separate auth, database, and object storage services.

**MLflow and DVC for ML workflow tracking.** The backend supports experiment logging via MLflow and data versioning via DVC. These are optional in development but recommended before retraining models in production so that model performance regressions are detectable.

## Role-based access and RLS

HCIMS AI has two roles: `admin` and `department_staff`. Role assignment happens at user creation time and is stored in the Supabase `profiles` table.

Supabase Row-Level Security policies translate the role into data access rules at the database level — not in application code. This means even a direct database query from an authenticated session cannot return rows that the user's role is not permitted to see.

| Capability | Admin | Department staff |
|---|---|---|
| Upload review CSV | Yes | No |
| View all tickets | Yes | No |
| View own-department tickets | Yes | Yes |
| Create staff accounts | Yes | No |
| Upload proof images | No | Yes |
| View appreciations | Yes | No |
| Access analytics dashboard | Yes | No |
| View activity logs | Yes | No |

<Note>
  The RLS policies must be applied in the target Supabase project. If you create a new Supabase project, run the migrations in `hcims-ai-main/supabase/` to provision the correct tables and policies before starting the application.
</Note>

## Proof image storage

Accepted proof images are stored in a Supabase Storage bucket named `proofs`. The backend uploads the image after the Nonescape validation passes. The frontend retrieves a signed URL from Supabase Storage to display the proof image in the admin ticket detail view.

The `PROOF_IMAGE_MAX_BYTES` environment variable (default `10485760` — 10 MB) enforces an upload size limit before the image reaches the Nonescape model. The detection threshold is controlled by `AI_IMAGE_CONFIDENCE_THRESHOLD` (default `0.45`).

## Service boundaries

| Service | Host | Default port |
|---|---|---|
| React frontend (Vite dev) | localhost | 8080 |
| FastAPI backend (Uvicorn) | localhost | 8001 |
| Supabase (cloud) | your project URL | 443 |

In production, the FastAPI backend should be deployed to a server or container platform, `VITE_API_URL` should point to the production API URL, and backend CORS should be restricted to the deployed frontend domain.
